# Integrating with GitHub Copilot

This guide covers how to use the voice profile and validation rules with GitHub Copilot for editorial review of content files.

## Prerequisites

- A completed voice profile at `${CLAUDE_PLUGIN_DATA}/profile.json`
- The human-voice plugin installed
- GitHub Copilot enabled on the repository

## Using the Voice Reviewer Agent

The repository includes a custom Copilot agent at `.github/agents/voice-reviewer.agent.md`. This agent reviews content files against the voice profile rules.

### Invoking the Agent

1. Open a content file in your editor
2. Open GitHub Copilot Chat
3. Select the "voice-reviewer" agent from the agent dropdown
4. Ask it to review the current file: "Review this content for voice compliance"

The agent checks for:

- Restricted characters (em dashes, smart quotes, emoji)
- Contractions
- Oxford comma usage
- AI writing anti-patterns (hedging, filler, meta-commentary)
- Voice profile deviations (tone, formality, structure)

### What the Agent Does Not Do

The voice-reviewer agent is editorial only. It does not:

- Conduct interviews or elicit voice profiles
- Run the scoring pipeline or NLP analysis
- Generate profiles or voice prompts
- Execute scripts or CLI tools

## Copilot Instructions

The file `.github/copilot-instructions.md` contains the full voice profile rules. Copilot reads this file automatically when working in the repository. All content generated by Copilot within this repository will follow the voice profile rules without explicit prompting.

### Updating After Re-Interview

If you run a new voice interview and the profile changes, update `.github/copilot-instructions.md` with the new rules from `${CLAUDE_PLUGIN_DATA}/voice-prompt.txt`. The voice profile rules in the instructions file are inlined, not referenced by path, so they require manual synchronization.

## CI Validation

The workflow at `.github/workflows/voice-validation.yml` runs automatically on pull requests that modify content files. It validates:

1. **Character restrictions**: Scans all markdown files in `docs/`, `content/`, `src/content/`, and `_docs/` for restricted characters
2. **Contractions**: Checks changed files for contraction usage (hard failure)
3. **Oxford comma**: Checks changed files for Oxford comma patterns (warning only)

### Workflow Trigger Paths

The workflow triggers on PRs that change files matching:

- `docs/**/*.md`
- `content/**/*.md`
- `src/content/**/*.md`
- `_docs/**/*.md`
- `README.md`
- `CONTRIBUTING.md`
- `CHANGELOG.md`

### Interpreting CI Results

- **Character restriction failures**: The validation script lists each violation with file path, line number, and the restricted character found. Replace the character according to the substitution table in the voice profile.
- **Contraction failures**: The check lists each contraction found with file and line number. Expand each contraction to its full form.
- **Oxford comma warnings**: These are flagged for manual review. Not all matches are true Oxford commas (compound clauses with "and" are valid). Review each match in context.
